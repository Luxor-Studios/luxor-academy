01Primer

Run MARS end-to-end on a stuck system you actually own

You have now seen the shape: parallel fan-out, epistemic constraint classification, systems-level metrics, five continuous architect gates. The MARS Quick Reference packages that whole arc as six concrete operations — and this module turns them back at you.

The six operations are commands you can call by name: /mars research <problem> for multi-domain parallel discovery, /mars synthesize <research> to integrate into a unified framework, /mars apply <blueprint> to map the framework to your real-world context, /mars optimize <system> to find leverage points, /mars validate <blueprint> to test against constraints, and /mars iterate <results> to learn and improve. Those six, in that order, are the full end-to-end run.

The Quick Reference is also explicit about when not to invoke MARS. Don't use MARS when: Single domain sufficient (use specialized agent instead); Simple tactical execution (use Task Relay); Pure research needed (use deep-researcher); Code implementation (use practical-programmer). MARS is for the multi-domain, systems-level, organizational-transformation class of problem — festival operations, manufacturing transformation, emergency-department throughput. If your stuck system fits that silhouette, it qualifies.

This module is the canonical last module of the quest. You pick one genuinely stuck system in your world — a process, an organization, a product line, a system that has fought every attempt at fixing it — and you run the six MARS operations on it end to end. You publish the analysis with sources cited. You get back a permalink you can hand to someone else. That is the sovereignty clause: you own what you ran, and you own what the run produced.

MARS ships as six named operations — research, synthesize, apply, optimize, validate, iterate — invoked in that order on a stuck system you own.
