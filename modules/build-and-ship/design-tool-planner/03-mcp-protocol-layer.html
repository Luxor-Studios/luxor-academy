01Primer

One tool interface to hold every visual API

The agents (Module 02) do not call Canva or Figma or Adobe directly. They call tools — objects that conform to the Model Context Protocol. Every tool, regardless of what API it wraps, has the same three-part shape: a name, a description that teaches the LLM when to reach for it, and an input_schema that constrains what arguments it will accept. The agent-builder guide makes this literal in lines 644-662: the MCPTool TypeScript interface has exactly those fields plus an execute(params) method that returns a ToolResult.

The payoff is fungibility. When you swap Canva for Figma, you do not edit the agent — you edit one tool file. The guide's example at lines 664-711 shows a CanvaAPI class implementing MCPTool: its execute method is a plain fetch() to https://api.canva.com/rest/v1/designs. Replace that URL and body with Figma's and you have a Figma tool that the exact same agent can use. The agent never knew there was a difference.

This is why the Universal Template generalises across Canva + NanoBanana, Figma + DALL-E, Adobe + Midjourney, and Sketch + Stable Diffusion. Each combination is a different set of tools — but the same set of agents, the same system prompts, the same decision matrix, the same quality gates. The canva-sdk-two-agent-spec.yaml formalises this at lines 109-136: the planner gets a short list of discovery tools (canva_list_templates, canva_get_brand_kit, analyze_design_requirements) while the executor gets the action tools (nanobanana_generate, canva_create_design, canva_add_elements, canva_export_design). Swap the prefix and the pattern still holds.

MCP is the seam. Swap the tool file, keep the agent. That is why one template covers Canva + NanoBanana, Figma + DALL-E, and every pair after.

### MCP Tool Interface

```typescript
interface MCPTool {
  name: string;
  description: string;
  input_schema: {
    type: 'object';
    properties: Record<string, any>;
    required: string[];
  };
  execute(params: object): Promise<ToolResult>;
}

interface ToolResult {
  success: boolean;
  data?: any;
  error?: string;
}
```

  






  
05Self-Check


  

    
1 — Per the MCPTool interface in lines 646–655, which field carries the description an agent reads to decide when to call the tool?

    

      
      
      
      
    

    
The interface declares description: string alongside name and input_schema. The agent selects tools by reading description text; name is only the identifier used when actually issuing the tool-use call. input_schema constrains arguments, not intent.

  


  

    
2 — In canva-sdk-two-agent-spec.yaml (lines 113–136), the planner and executor agents are given different tool lists. Which tool is assigned to the executor and NOT to the planner?

    

      
      
      
      
    

    
The spec at line 127 lists nanobanana_generate as an executor tool (it generates assets — a tactical action). The planner's tools (lines 113–116) are read-only discovery tools: list templates, fetch brand kit, analyze requirements. The split mirrors the strategic-vs-tactical division from Module 02.

  


  

    
3 — The CanvaAPI example at lines 667–677 declares name = 'canva_create_design' and an input_schema requiring which two fields?

    

      
      
      
      
    

    
Lines 670–677: the input_schema declares properties template_id and title, and required: ['template_id', 'title']. These are the two arguments the LLM must supply on every tool call. The implementation then posts them to the Canva REST endpoint at line 684.
