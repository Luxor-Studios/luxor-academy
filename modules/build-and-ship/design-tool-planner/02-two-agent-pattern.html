01Primer

Planner plus Executor: the validated shape

A single Claude agent trying to both decide what to build and do the building is a common first instinct — and a common first failure mode. The Agent Builder Guide (lines 218-227) flags the alternative: split the work. A Planning Agent (strategic, 5-10 seconds, 2048-token budget) parses intent, picks templates, and emits a DesignPlan as structured JSON. An Execution Agent (tactical, 20-25 seconds, 8192-token budget) receives that plan and only has to follow it, generating assets and assembling the design via MCP tools.

This is the pattern the guide stars with ⭐ RECOMMENDED — and the only pattern with a validated quality number: 9.5/10 on the CANVA-CODE run, the highest step-score in the whole workflow. Total wall time came in around 28 seconds against a 30-second target; token cost settled at about $0.32 per design against a $0.50 ceiling; first-try quality hit 92%; brand consistency hit 99%+. Those are not aspirational figures — they are line-517 of the guide, next to the word Validated.

Why does the split work? Three reasons stated in docs/ARCHITECTURE.md lines 74-78: clear separation of concerns (what vs how), per-role optimization (different system prompts, different token budgets, different tools), and better error recovery (a failed execution does not corrupt the plan). Iterations become cheap — you refine the design without re-planning it. For 3-10 API calls per request, this is the pattern. Fewer calls, go single-agent. More than ten, go multi-agent. In between — which is most real projects — the two-agent pattern is the quiet winner.

Two agents, two budgets, two jobs. The Planning Agent decides (5-10s, strategic); the Execution Agent builds (20-25s, tactical). Validated at 9.5/10 on CANVA-CODE.
