Dual-Theme CSS via Variables

// Tier: Initiate · Source: content/sources/barque/barque/core/themes.py

01Primer

One stylesheet, two PDFs

BARQUE's headline feature is a single Markdown input that produces both a light-theme and a dark-theme PDF. Look at the generator and you will not find two parallel stylesheets. You will find one, parameterised by CSS custom properties declared on :root.

The logic lives in barque/core/themes.py. The ThemeProcessor.generate_css(theme) method takes a string — "light" or "dark" — pulls the corresponding colour dict from BarqueConfig, and emits a CSS block whose :root declares --bg-primary, --text-primary, --accent-color, --code-bg and friends. Every downstream rule (body, h1, pre, blockquote, table) references those variables rather than hard-coded hex. Change the six colour tokens, render again, and the entire document retypes itself in the new palette.

This is the same pattern atlas/tokens/tokens.css uses for this very course — --gold-300, --navy-050, --surface-0, swapped under a [data-theme="parchment"] selector. CSS variables are a runtime indirection: the browser (or in BARQUE's case, WeasyPrint) resolves the token at paint time, so two themes never mean two stylesheets. They mean one stylesheet, two inputs.

Dual themes are not two stylesheets — they are one stylesheet and two palettes, wired together through :root custom properties.

:root {{
    /* Theme Colors */
    --bg-primary: {bg_color};
    --bg-secondary: {self._lighten_darken(bg_color, 0.05 if theme == "dark" else -0.03)};
    --text-primary: {theme_data.get('text', '#000000')};
    --text-secondary: {self._lighten_darken(theme_data.get('text', '#000000'), -0.2 if theme == "light" else 0.15)};
    --border-color: {theme_data.get('border', '#e0e0e0')};
    --accent-color: {theme_data.get('accent', '#2563eb')};
    --code-bg: {code_bg_enhanced};
    --code-border: {code_border};
    --code-text: {theme_data.get('code_text', theme_data.get('text', '#000000'))};
    --link-color: {theme_data.get('accent', '#2563eb')};
    --link-hover: {self._lighten_darken(theme_data.get('accent', '#2563eb'), -0.1)};

    /* Typography - Enhanced for readability */
    --font-family: {self.config.font_family};
    --base-font-size: {base_font};
    --line-height: {self.config.line_height};
    --max-width: {self.config.max_width};
}}

05Self-Check

1 — In barque/core/themes.py, where does ThemeProcessor.generate_css look up the colour values for the chosen theme?
_get_theme_data(theme) (themes.py lines 354–361) returns self.config.light_theme or self.config.dark_theme — both Dict[str, str] defined on BarqueConfig (config.py lines 29–45). The CSS template interpolates those values into :root.
2 — Per barque/core/config.py, what is the default accent colour for the light_theme dict?
config.py line 33: "accent": "#2563eb". The dark theme's accent (line 42) is "#60a5fa". Those two hex values are the only thing distinguishing light from dark at the accent-colour level — everything downstream is just var(--accent-color).
3 — A body rule inside the generated CSS references background-color: var(--bg-primary);. Why does this one rule produce two different backgrounds in the two PDFs?
var(--bg-primary) is resolved at paint time against the :root declaration, which was emitted with a different hex value per theme (themes.py line 56). The rule is identical in both stylesheets. The palette is not.
