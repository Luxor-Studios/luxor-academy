01Primer

Why BARQUE insists on source venv/bin/activate

BARQUE's first instruction is not pip install — it is source venv/bin/activate. That line is not a suggestion. It is load-bearing.

Many Python CLIs you install through Homebrew (including weasyprint, which BARQUE shells out to for PDF rendering) ship with an absolute shebang pointing at a specific Python interpreter path. When macOS upgrades Python underneath you — say 3.12 to 3.13 — the binary at that path disappears, and the CLI errors out with bad interpreter: no such file or directory. The tool is not broken. Its pointer moved.

A virtual environment sidesteps this entirely. source venv/bin/activate prepends venv/bin to your PATH, so every call to python and weasyprint resolves to the venv-local binary first. The venv writes shebangs that resolve relative to the activated shell, not to a Homebrew path that may have already vanished.

BARQUE's QUICK-START.md enshrines this in two lines: cd into the repo, then source venv/bin/activate, then pip install -e .. Skip the middle line and nothing works. Run the middle line and everything does.

An absolute shebang is a fragile pointer. source venv/bin/activate replaces it with a relocatable one by editing PATH.

05Self-Check

1 — BARQUE's QUICK-START.md lists three installation steps in order. What is the second step, right before pip install -e .?
QUICK-START.md §Installation (lines 10–14) shows the exact order: cd into the repo, then source venv/bin/activate, then pip install -e .. The venv activation is non-negotiable — it is what makes the editable install resolve interpreter paths correctly.
2 — Why does activating the venv change where python3 resolves?
The activate script prepends venv/bin to the PATH environment variable. Shell lookup walks PATH left-to-right and returns the first match, so venv/bin/python3 is found before any Homebrew or system Python.
3 — BARQUE's QUICK-START.md Troubleshooting section tells you to run source venv/bin/activate before which check?
QUICK-START.md lines 202–206: before python -c "import weasyprint". The import only succeeds if the venv is active, because weasyprint was pip-installed into the venv, not globally.
