Prompt Caching for Real

// Tier: Initiate · Source: claude-agent-sdk-multiplatform

01Primer

The cache_control annotation is a one-line cost lever

Every request to the Messages API sends your full system prompt and tool schemas over the wire and re-processes them on Anthropic's side. For an agent with a 1,500-token system prompt and four elaborate tool schemas, that is the same ~4 KB of context being tokenised on every turn. Prompt caching trades that for a cache read at roughly a tenth of the input-token cost, provided the content is marked with cache_control.

The typescript/GUIDE.md caching example on lines 686-709 shows the shape. Instead of passing system as a bare string, you pass it as an array of structured blocks: system: [{ type: "text", text: systemPrompt, cache_control: { type: "ephemeral" } }]. That cache_control: { type: "ephemeral" } annotation tells the server to write a 5-minute cache entry keyed on the exact block content. The first request pays full input cost and populates the cache; every subsequent request whose prefix hashes identically reads from the cache and pays the discounted cache_read_input_tokens rate.

Three things to know before you turn this on. First, the cache is prefix-sensitive: cached blocks must appear in the same order, earlier in the request, than the non-cached blocks. Second, you observe cache activity via the response's usage object — cache_creation_input_tokens on the first hit, cache_read_input_tokens on each reuse. Those two counters are how you prove the optimisation is actually firing. Third, tool schemas benefit the most: a large schema block marked with cache_control saves the same tokens every loop iteration, which matters exactly as much as your agent loops. Module 04's loop is where caching pays off.

cache_control: { type: "ephemeral" } on a structured system or tools block is a one-line cost cut. The proof lives in response.usage.cache_read_input_tokens — if that number is nonzero on turn two, the cache is real.

04Artifact

typescript/GUIDE.md · lines 686–709 view vendored source → 
async function sendWithCache(
  client: Anthropic,
  systemPrompt: string,
  userMessage: string
): Promise<string> {
  const response = await client.messages.create({
    model: "claude-sonnet-4-5-20250929",
    max_tokens: 1024,
    system: [
      {
        type: "text",
        text: systemPrompt,
        cache_control: { type: "ephemeral" },
      },
    ],
    messages: [{ role: "user", content: userMessage }],
  });

  return response.content[0].text;
}

// Subsequent requests with same system prompt will use cache

05Self-Check

1 — typescript/GUIDE.md lines 686–709 shows how to pass the system prompt for caching. What structural change must you make relative to the non-cached version?
GUIDE.md lines 694–701: the system field becomes an array of structured content blocks. Each block may carry a cache_control annotation. Only the structured form accepts cache_control; the plain-string shortcut from Module 02 does not.
2 — What value is passed to the cache_control field on GUIDE.md line 699?
GUIDE.md line 699: cache_control: { type: "ephemeral" }. ephemeral is the currently supported cache type — a short-lived (~5 minute) cache entry keyed on exact block content. There is no persistent variant in the block-level cache_control API, and a bare boolean is rejected.
3 — You wire up sendWithCache exactly as GUIDE.md shows and call it twice in a row with the same systemPrompt. On the second response, which usage field tells you the cache fired?
The usage object on the response exposes cache_creation_input_tokens on the first (miss) request and cache_read_input_tokens on subsequent (hit) requests. GUIDE.md calls this behaviour out in the comment on line 708: "Subsequent requests with same system prompt will use cache." The counter is how you verify it actually fired — a stubborn zero means something in the prefix differed between calls.
