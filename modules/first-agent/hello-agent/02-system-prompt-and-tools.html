System Prompt + Your First Tool

// Tier: Initiate · Source: claude-agent-sdk-multiplatform

01Primer

Two knobs: who the agent is, and what it can do

A raw messages.create call gives Claude a user turn and nothing else. Two small fields in that call turn it into something worth calling an agent. The first is system — a string (or structured block) that sets the agent's identity, constraints, and tone before any user turn. The TypeScript GUIDE shows it on lines 184-189: pass system: "You are a helpful TypeScript expert. Provide concise, practical answers." and every reply now inherits that character. The system field is the only persona the agent ever has; the user cannot override it mid-conversation.

The second field is tools. A tool is not code — it is a schema Claude reads to decide whether and how to call a function you own. The GUIDE defines one on lines 319-354: a get_weather tool with a name, a description, and an input_schema (JSON Schema describing the shape of location and unit). You hand the array of tool schemas to messages.create and Claude, if it judges the tool relevant, responds with a tool_use content block containing a concrete input object.

The trick is that this module stops one step short of actually executing the tool. That closing step — running the function, returning a tool_result, looping — is the subject of Module 04. Here you only learn to declare the schema and watch Claude choose to call it. That half alone is the entire contract: description quality drives call quality, and schemas are the agent's only view into your code.

system is who the agent is. tools is what the agent can choose to reach for. Both are declared on the same messages.create call; neither exists without it.

  {
    name: "get_weather",
    description: "Get the current weather for a location",
    input_schema: {
      type: "object",
      properties: {

          type: "string",
          description: "City name, e.g., 'San Francisco'",
        },
        unit: {
          type: "string",
          enum: ["celsius", "fahrenheit"],
          description: "Temperature unit",
        },
      },
      required: ["location"],
    },

05Self-Check

1 — In typescript/GUIDE.md lines 319–354, each tool object has exactly three required top-level fields before the schema is complete. Which trio?
GUIDE.md lines 321–323 (get_weather) and lines 341–343 (calculate) show the same three fields. name is the identifier Claude returns in tool_use.name; description drives whether Claude decides to call it; input_schema is the JSON Schema the returned tool_use.input must match.
2 — typescript/GUIDE.md shows system prompts on lines 184–189. What does the system field define?
GUIDE.md lines 184–198: system: "You are a helpful TypeScript expert. Provide concise, practical answers." fixes the agent's role. The user turn (line 188: messages: [{ role: "user", content: userMessage }]) runs under that system prompt. The user cannot override system mid-conversation.
3 — A tool's input_schema uses JSON Schema. The get_weather schema on GUIDE.md lines 324–338 lists location and unit under properties, but required: ["location"]. What does this mean for Claude's call?
GUIDE.md line 337: required: ["location"]. Only location is mandatory. unit (lines 331–335) is optional and constrained to the enum ["celsius", "fahrenheit"] when present. JSON Schema's required list is the exact same contract as in any other JSON Schema consumer.
