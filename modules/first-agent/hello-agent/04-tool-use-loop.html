Close the Tool-Use Loop

// Tier: Initiate · Source: claude-agent-sdk-multiplatform

01Primer

The three states of an agent loop: end_turn, tool_use, iterate

Module 02 taught you to declare a tool schema. Module 04 teaches you what the agent actually does when Claude decides to call it. The answer is a tight while (true) loop with exactly two exit/continuation branches, keyed on response.stop_reason.

The typescript/GUIDE.md loop on lines 388-440 is the canonical shape. You initialize messages with the user's turn. Inside the loop you call messages.create with the tools array. The response comes back with a stop_reason. If it is "end_turn", Claude has produced the final text — return it and exit. If it is "tool_use", the response's content array contains one or more tool_use blocks, each with a name, an id, and an input object that validates against your schema. You append the assistant's message verbatim (messages.push({ role: "assistant", content: response.content })), then execute the tool function (a lookup by block.name), then append a user message whose content is an array of tool_result blocks (one per tool call, correlated by tool_use_id to the original id), then loop. Claude sees the tool results on the next iteration and either calls another tool or returns end_turn.

The two loop invariants that matter: (1) every tool_use must be answered by a tool_result with the same tool_use_id — the SDK rejects the next request otherwise; (2) the assistant turn you push back must be response.content unchanged, not a re-serialisation. Break either invariant and the loop either 400s immediately or silently drops context.

The agent loop is a state machine with two labels: end_turn exits, tool_use executes-and-continues. The tool_result block ties the loop back to the tool_use block via tool_use_id — that ID is the single thread that keeps the conversation coherent.

async function runWithTools(userMessage: string): Promise<string> {
  const messages: Anthropic.Messages.MessageParam[] = [
    { role: "user", content: userMessage },
  ];

  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-5-20250929",
      max_tokens: 4096,
      tools,
      messages,
    });

    console.log("Stop reason:", response.stop_reason);

    // If Claude is done, return the final response
    if (response.stop_reason === "end_turn") {
      const textContent = response.content.find((c) => c.type === "text");
      return textContent?.text || "";
    }

    // If Claude wants to use a tool
    if (response.stop_reason === "tool_use") {
      // Add Claude's response to messages
      messages.push({ role: "assistant", content: response.content });

      // Process each tool use
      const toolResults: Anthropic.Messages.ToolResultBlockParam[] = [];

      for (const block of response.content) {
        if (block.type === "tool_use") {
          console.log(`Calling tool: ${block.name} with`, block.input);

          const toolFunction = toolFunctions[block.name];
          if (!toolFunction) {
            throw new Error(`Unknown tool: ${block.name}`);
          }

          const result = await toolFunction(block.input);

          toolResults.push({
            type: "tool_result",
            tool_use_id: block.id,
            content: JSON.stringify(result),
          });
        }
      }

      // Add tool results to messages
      messages.push({ role: "user", content: toolResults });
    }
  }
}

05Self-Check

1 — In typescript/GUIDE.md lines 388–440, what value of response.stop_reason causes runWithTools to return its final answer and exit the loop?
GUIDE.md lines 404–407: if (response.stop_reason === "end_turn") returns the text block. "tool_use" (line 410) is the continue branch — it means Claude wants to call a tool and expects you to loop. "max_tokens" would mean the cap you set on line 396 was hit; the example does not handle that case.
2 — On GUIDE.md line 430, each tool_result block carries tool_use_id: block.id. What happens if you drop that field or use a different value?
GUIDE.md line 430: tool_use_id: block.id is the correlation key. The API pairs every tool_use block (which has an id) with exactly one tool_result block (whose tool_use_id matches). A missing or mismatched ID yields a 400 from the API on the next iteration.
3 — Why does GUIDE.md line 412 push response.content verbatim into messages as the assistant turn, instead of just pushing the text?
GUIDE.md line 412: messages.push({ role: "assistant", content: response.content }). The assistant's content includes the tool_use blocks verbatim. On the next turn, Claude reads its own previous tool_use and the user's paired tool_result to know what happened. Replacing response.content with just the text strips the tool_use block and breaks the pairing.
