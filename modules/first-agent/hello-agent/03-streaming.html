Stream the Response

// Tier: Initiate · Source: claude-agent-sdk-multiplatform

01Primer

From a blocking promise to a token-by-token async iterator

Without streaming, messages.create is one round-trip: you await the whole reply and print it at the end. For a CLI this is the wrong shape. The user stares at a blinking cursor for three seconds, then the entire reply lands in one paste. Streaming fixes that by returning an async iterable of events — you consume them with for await (const event of stream) and render each token as it arrives.

The mechanics are a single boolean flip. typescript/EXAMPLES.md lines 137-161 show the whole thing: pass stream: true to messages.create and the SDK returns an AsyncIterable<MessageStreamEvent> instead of a single Message. You iterate the stream; you filter for events where event.type === "content_block_delta" and event.delta.type === "text_delta"; you process.stdout.write(event.delta.text) for each one. That is the complete streaming loop.

Three things move from implicit to explicit once you stream. First, event types — message_start, content_block_start, content_block_delta, content_block_stop, message_delta, message_stop — and you only care about content_block_delta for rendering text. Second, token usage only finalises in the message_delta event at the end of the stream, so cost tracking attaches to that event specifically. Third, errors can arrive mid-stream, so the try/catch goes around the for await loop, not around the initial create call. Everything else — the model ID, the messages, the max_tokens cap — is identical to the non-streaming version from Module 01.

Streaming is not a different API. It is the same messages.create with stream: true, returning an async iterable whose content_block_delta events carry the text you render.

04Artifact

typescript/EXAMPLES.md · lines 137–163 view vendored source → 
async function streamingChat(message: string): Promise<void> {
  const client = new Anthropic({
    apiKey: process.env.ANTHROPIC_API_KEY!,
  });

  console.log("Claude: ");

  const stream = await client.messages.create({
    model: "claude-sonnet-4-5-20250929",
    max_tokens: 1024,
    messages: [{ role: "user", content: message }],
    stream: true,
  });

  for await (const event of stream) {
    if (
      event.type === "content_block_delta" &&
      event.delta.type === "text_delta"
    ) {
      process.stdout.write(event.delta.text);
    }
  }

  console.log("\n");
}

streamingChat("Write a short story about a robot learning to paint");

05Self-Check

1 — In typescript/EXAMPLES.md lines 137–163, which single field turns the messages.create call into a streaming call?
EXAMPLES.md line 148: stream: true. That one boolean flips the return type from a Message to an AsyncIterable<MessageStreamEvent>. Everything else in the call — model, max_tokens, messages — is identical to the non-streaming path from Module 01.
2 — The streaming loop filters events with two conditions (EXAMPLES.md lines 152–155). What is the filter looking for?
EXAMPLES.md lines 152–155 filter for content_block_delta events whose delta.type is text_delta. Those are the events that carry renderable text. message_start, content_block_start, content_block_stop, message_delta, and message_stop also arrive, but the CLI renderer ignores them.
3 — After the filter matches, what does the example do with the token on EXAMPLES.md line 156?
EXAMPLES.md line 156: process.stdout.write(event.delta.text). write (unlike console.log) does not add a trailing newline, which is what you want for token-by-token rendering — each text_delta is a fragment, and fragments concatenate into a word. The trailing console.log("\n") on line 160 adds the single final newline after the stream completes.
