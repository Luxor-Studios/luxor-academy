L5 — Effect Polymorphism

// Tier: Master · Source: fstar-framework · Level 5/7

01Primer

Effects as types — stateful code, still verified

At Level 5 the type system gains a whole new axis. Alongside int, list a, and refinements, every computation now carries an effect label — Tot for total, Dv for divergent, ST for stateful, GTot for erasable ghost code. The effect is part of the type. F* can talk about state, exceptions, and divergence inside the same logic that proved clamp and fibonacci, and it can mix them through effect polymorphism and lifting.

The canonical L4/L5 example in the fstar-framework is swap, written in the ST (state) effect with an explicit Hoare triple:

let swap (#a: Type) (r1 r2: ref a) : ST unit
  (requires (fun h -> True))
  (ensures (fun h0 _ h1 ->
    sel h1 r1 == sel h0 r2 &&
    sel h1 r2 == sel h0 r1 &&
    modifies (only r1 ++ only r2) h0 h1))
=
  let tmp = !r1 in
  r1 := !r2;
  r2 := tmp

Three new things are loud in that signature. Requires names the precondition on the incoming heap h. Ensures names the postcondition as a relation between pre-heap h0, return value, and post-heap h1. And modifies names a frame — the set of references that can change. F* requires you to prove that nothing outside the frame is touched, which is how effect polymorphism composes: if f : ST _ _ _ (modifies S0) and g : ST _ _ _ (modifies S1), then sequencing them gives a combined modifies (S0 ++ S1), and the rest of the heap is provably untouched.

The skeleton extrapolates. ST is one effect in a lattice — Tot ⊂ Ghost ⊂ ML ⊂ Dv, with ST sitting orthogonally over the heap. Write your algorithm once over an effect variable; instantiate it for Tot when you want a pure extraction, for ST when you need real memory. Same code, different effect, same proofs.

At L5 the effect is part of the type. ST unit (requires ...) (ensures ...) is a Hoare triple embedded in F*'s type system, and the modifies clause is the frame condition that makes effect polymorphism compositional.

05Self-Check

1 — In the swap signature, sel h1 r1 == sel h0 r2 appears in the ensures clause. What do h0 and h1 refer to, and what is sel?
h0 is the **pre-heap** (state before the call), h1 is the **post-heap** (state after), and sel h r reads the value stored at reference r in heap h. The ensures clause is therefore the relational statement 'the value of r1 after the call equals the value r2 held before the call' — the core correctness of a swap.
2 — The modifies (only r1 ++ only r2) h0 h1 clause is the **frame condition**. Why does F* require it?
Effect polymorphism is only compositional if you can talk about what a computation *does not* touch. The frame condition names the set of references {r1, r2} that swap may write to, and F* then proves every other reference in the heap is unchanged — so when you sequence swap r1 r2; do_something_with r3, the solver knows r3's value survived intact.
3 — The framework document places ST in a lattice of effects. What is the practical consequence of Tot ⊂ ST in that lattice?
Every total (pure) computation can be **lifted** into the stateful effect for free — the heap is simply unchanged. That is what lets you write a library function once in Tot and call it safely from ST code. The subsumption is one direction only: you cannot silently demote an ST computation to Tot, because that would lose the heap reasoning.
