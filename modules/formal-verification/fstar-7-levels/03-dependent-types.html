L3 — Dependent Types

// Tier: Master · Source: fstar-framework · Level 3/7

01Primer

Types that depend on values — termination proofs as first-class

Level 3 is where the type system starts to compute. A refinement type lets a type depend on a predicate; a full dependent type lets a type depend on a value — a length, a bound, a proof term. The first step onto that ladder is the humble decreases clause: a value-level expression that appears inside a function's type and that F* uses to prove termination.

Fibonacci is the clean case, because it recurses twice:

let rec fibonacci (n:nat) : nat
  decreases n
=
  if n = 0 then 0
  else if n = 1 then 1
  else fibonacci (n - 1) + fibonacci (n - 2)

decreases n says: "at every recursive call, the value of n gets strictly smaller in the natural-number ordering." F* checks this by hand: fibonacci (n - 1) needs n - 1 < n (true when n >= 2, which is exactly the branch we are in); fibonacci (n - 2) needs n - 2 < n (same). Both succeed and F* accepts the function under the Tot (total) effect.

Three quiet things just happened. First, n:nat is itself a refinement — nat = x:int{x >= 0} — so the dependent type nat -> nat is carrying a value-level invariant. Second, the metric n is a term of nat; it is literally a value that the type system consults. Third, the presence of decreases is what promotes the definition from possibly non-terminating (Dv effect) to total (Tot effect). Remove it and F* rejects the function outright — not because it is wrong, but because F* refuses to believe in termination without evidence.

decreases n is a dependent type feature hiding in plain sight: a value-level term that becomes part of the function's type, and the gate that lets a recursive definition live in the total (Tot) effect.

05Self-Check

1 — What does the decreases n clause on fibonacci promise, and how does F* check it?
It promises that at every recursive call site, the value of the argument n is strictly smaller than it was at the caller, measured in the natural-number ordering. F* discharges this by proving n-1 < n and n-2 < n under the branch context n >= 2 — both follow trivially once the base cases have been peeled off.
2 — If the decreases clause is removed from fibonacci, which effect does F* fall back to?
Per the Week 2 solutions answer key: without a termination argument F* will not admit the function into Tot (total). You can explicitly annotate it with Dv (divergence) — e.g., let rec bad_fib (n:int) : Dv int = bad_fib n — to tell F* 'I know this might not terminate, accept it anyway.' The clause is what promotes a definition from Dv to Tot.
3 — Why does F* accept fibonacci : nat -> nat but refuse fibonacci : int -> int under the same body?
nat = x:int{x >= 0} is already a refinement. Inside the recursive branch n >= 2, the arguments n-1 and n-2 are both non-negative, so they still inhabit nat. If the type were int -> int, the call fibonacci (n-1) in an expansion of fib 1 would hand -1 to itself — and the solver has no way to refute that unless the type carries the >= 0 invariant. The dependent refinement is what keeps the metric honest.
