01Primer

Your first F* proof: let Z3 close the gap

A "proof" in F* does not always look like a proof. At Level 1 of the 7-level ladder, a proof looks like a type. You write a function, you annotate its argument with a refinement predicate, and when F* typechecks the file it hands the resulting logical obligation to the Z3 SMT solver. If Z3 returns unsat on the negation of your postcondition, F* accepts the function as verified. You never wrote theorem or QED. You wrote a type.

The canonical Week 1 example is safe division. int / 0 is undefined, so we carve out the subset of integers that are not zero and call that nonzero:

type nonzero = x:int{x <> 0}

let safe_divide (a:int) (b:nonzero) : int = a / b

Two things happen here. First, the refinement {x <> 0} is syntactic sugar for a predicate subtype: nonzero contains every integer x for which x <> 0 holds. Second, any call site that passes 0 where nonzero is expected fails at compile time with Could not prove refinement: 0 <> 0. The solver attempted the check and Z3 handed back sat with the witness x = 0. That counterexample is the error.

Extracted to OCaml, the refinement evaporates — the shipped code is just let safe_divide a b = a / b, no runtime guard, because the proof already ran at build time. That is the whole pedagogy of L1: write tight types, let the SMT solver discharge them, ship boring code that is provably safe.

At L1, the proof IS the type. A refinement {x <> 0} is a predicate subtype — Z3 discharges the obligation at typecheck time and the check erases on extraction.

05Self-Check

1 — In the definition type nonzero = x:int{x <> 0}, what exactly does the brace-enclosed predicate do?
It defines a **predicate subtype** (also called a refinement type): nonzero is the set of integers x for which x <> 0 holds. Membership is checked at typecheck time by Z3, not at runtime. Extraction erases the refinement, so the OCaml output is just int.
2 — A learner writes let test_fail1 = safe_divide 10 0 and F* refuses to compile. Which error message is the one F* actually emits, per week_01_all_solutions.fst?
The inline comment in the solutions file names it exactly: Could not prove refinement: 0 <> 0. F* handed Z3 the obligation 0 <> 0; Z3 returned sat (the witness is x = 0 itself), so the refinement is unprovable and the call is rejected at compile time.
3 — After F* extracts safe_divide to OCaml, what happens to the nonzero refinement at runtime?
Refinement types are **erased** during extraction. The OCaml code is just let safe_divide a b = a / b with no runtime check — F* already proved safety at compile time, so there is nothing left to check when the program runs. Week 1 teaching notes call this out as the Level-1 payoff.
