L2 — Refinement Types

// Tier: Master · Source: fstar-framework · Level 2/7

01Primer

Carry the invariant in the type, not in a comment

Level 2 tightens the screw. Now the refinement is compound — multiple conjoined predicates that describe a non-trivial invariant — and the signature of the function returns a refined type, not just accepts one. The question is no longer "can I refuse bad input?" but "can I prove the output is always good?"

The canonical Week 1 exercise is clamp. The spec: given a range [min, max] and an input x, return a value provably inside the range. The F* type is where the invariant lives:

let clamp (min:int) (max:int{max >= min}) (x:int)
  : y:int{y >= min && y <= max} =
  if x < min then min
  else if x > max then max
  else x

Read the return type aloud: "returns some integer y such that y >= min && y <= max." That is not a comment — it is a proof obligation that F* must discharge for every branch of the body. And here is the beautiful part: F* tracks control flow. In the if x < min then min branch, the solver knows x < min holds; it must prove min >= min && min <= max, and the second conjunct follows from the precondition max >= min. In the final else x branch, the solver has accumulated x >= min && x <= max from the failed guards and closes the proof for free.

Remove the precondition max >= min and the first branch collapses — F* reports could not prove post-condition: min <= max, because the assumption that makes it true is gone. The invariant is the type, and the type is binding.

At L2, the return type *is* the postcondition and F* tracks control-flow facts through each branch to discharge it. A missing precondition makes the proof collapse at the first branch that needed it.

02Visual

A three-branch decision tree for clamp(min, max, x). The root is labeled with the signature y:int{y >= min && y <= max}. Branch 1 if x < min returns min and the annotation reads context: x < min; goal: min >= min && min <= max; discharged by precondition max >= min. Branch 2 else if x > max returns max with a symmetric annotation. Branch 3 else x returns x with context: x >= min && x <= max (from negated guards); goal holds directly. A side note shows that removing the precondition collapses branch 1 into a compile error.

// Part B: Positive refinement

// Part C: Bounded integers
type bounded100 = x:int{x >= 0 && x < 100}

05Self-Check

1 — In clamp's final branch else x, how does F* prove the return value satisfies y >= min && y <= max?
F* performs **control-flow analysis**. The else branch is only reached when both prior guards failed, so the solver knows not (x < min) and not (x > max), i.e., x >= min && x <= max — which is exactly the postcondition. The solutions file names this out explicitly: 'F* proves: directly from control flow context!'
2 — Why does clamp's signature need the precondition max:int{max >= min}?
Without it, branch 1 if x < min then min must still prove min <= max — and with no assumption relating min and max, the solver cannot discharge that conjunct. The teaching notes state the exact error: 'Could not prove postcondition: min <= max'. The precondition closes the gap.
3 — increment_wrap (x:bounded100) : bounded100 returns if x = 99 then 0 else x + 1. An alternative implementation uses (x + 1) % 100. What is the pedagogical trade-off the Week 1 solutions file names between these two?
Per the instructor notes, the modulo version is more concise but shifts the work to the SMT solver — it must prove (x + 1) % 100 < 100 holds across the whole domain. The if-then-else version makes the two cases explicit so control-flow analysis closes the proof directly. Both verify; the difference is clarity vs. compactness.
