L4 — Indexed Types

// Tier: Master · Source: fstar-framework · Level 4/7

01Primer

Lists indexed by length — non-structural recursion, tamed

Level 4 starts to pay. An indexed type family attaches a value — typically a length, a height, or a phase — to each instance of a data structure, so shape errors fail at the type level. The Week 2 mini-project is the classic example: verified merge sort, whose recursion splits its argument into two halves and is therefore not structural on tl. F* cannot guess the metric; you must hand it one.

The predicates are indexed by a list. The sortedness check walks pairs:

let rec sorted (xs:list int) : bool =
  match xs with
  | [] -> true
  | [x] -> true
  | x :: y :: rest -> x <= y && sorted (y :: rest)

And termination of merge_sort itself relies on the length-indexed view of the list. The mini-project spells it out in bold: the recursion is not structural, so you must supply decreases (length xs). F* then verifies that split produces two strictly-shorter sublists — an obligation on length, not on the list constructor — and closes the proof. The index (length) is the termination metric; the data (the list) is the computational content; F* keeps them separate and synchronised through the type.

That is the L4 shift in attitude. At L3 you proved termination of fibonacci on a raw nat. At L4 you prove it on a measure over the data — a function from your structure into a well-founded ordering. The indexed family is the pattern: length-indexed vectors, height-indexed trees, phase-indexed state machines, merge-sort-indexed-by-list-length. Once you can see the index, you can see why the recursion terminates, and F* will see it with you.

At L4 the termination metric is a *measure* over the data, not the data itself: decreases (length xs) turns non-structural divide-and-conquer into a termination proof F* can check.

let rec split (#a:Type) (xs:list a) : (list a * list a) =
  match xs with
  | [] -> ([], [])
  | [x] -> ([x], [])
  | x :: y :: rest ->
      let (left, right) = split rest in
      (x :: left, y :: right)

val merge: xs:list int -> ys:list int -> list int
val merge_sort: xs:list int -> list int

val merge_sorted:
  xs:list int -> ys:list int ->
  Lemma (requires sorted xs /\ sorted ys)
        (ensures sorted (merge xs ys))

05Self-Check

1 — The mini-project README says the recursion in merge_sort is NOT structural. What does that mean, and why does it matter for termination?
Structural recursion recurses on an immediate sub-constructor of the input — e.g., tl of a hd :: tl match. merge_sort recurses on split xs, which is a function call, not a constructor. F* cannot see 'smaller' just by looking at the pattern, so without an explicit decreases clause over a well-founded measure it refuses to admit the definition into the total effect.
2 — Which decreases clause does the mini-project tell you to use for merge_sort?
Per the mini_project_merge_sort.fst hint block: 'What metric decreases with each recursive call? Answer: length of the list! Use: decreases (length xs). F* will verify that split produces shorter sublists.' The length of the list is the index that shrinks; the list itself is not.
3 — The lemma merge_sorted is stated with requires sorted xs /\ sorted ys and ensures sorted (merge xs ys). What role does this lemma play in proving that merge_sort produces sorted output?
It is a **helper lemma**. The main theorem (merge_sort produces a sorted list) is proved by induction on length xs and uses merge_sorted at the inductive step: after the two recursive calls return sorted sublists, merge_sorted is what lets F* conclude that merging them preserves the invariant. The indexed-family pattern composes: prove per-index invariants, then compose them.
