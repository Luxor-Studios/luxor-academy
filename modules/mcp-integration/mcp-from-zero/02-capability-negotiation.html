01Primer

The handshake that decides what each side can say

Nothing useful travels over MCP until both ends have agreed on what each of them speaks. That agreement happens in the very first message pair — the initialize handshake — and the word to watch for is capabilities.

A client opens the transport, sends an initialize request advertising its protocol version and its own capability bag, and waits. The server answers with its matching capability bag. Only the capabilities that appear in both bags are valid for the rest of the session. tools, resources, prompts, logging, sampling — each is a separate switch, and the switches are set once at the top of the conversation.

The mcp-categorical skill makes this concrete in the simplest possible TypeScript server. Look at the new Server(...) line: the second argument is a literal capability declaration, { capabilities: { tools: {} } }. That object is the payload the server ships back during initialize. Writing { tools: {} } tells the client: I speak tools; I do not speak resources or prompts. If the client later sends a resources/list request, the server is free (and correct) to reject it.

That means your first architectural decision as a server author is not which tool to register. It is which primitive families you will claim in this capability object. The handshake is a cut — every promise your server will keep, and every promise it will refuse to keep, is declared in that one literal.

initialize is not a greeting. It is a cut. The capabilities object is a contract the server signs, once, before any real request arrives.

## Basic MCP Server (TypeScript)

```typescript
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  CallToolRequestSchema,
  ListToolsRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";

// Create server (category of tools)
const server = new Server(
  { name: "categorical-server", version: "1.0.0" },
  { capabilities: { tools: {} } }
);

// Define tool (morphism)
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: "analyze",
      description: "Analyze input with categorical structure",
      inputSchema: {
        type: "object",
        properties: {
          input: { type: "string" },
          category: { type: "string" }
        },
        required: ["input"]
      }
    }
  ]
}));

05Self-Check

1 — In the mcp-categorical TypeScript example, which literal declares the server's capabilities to the client?
skills/integrations/mcp-categorical/SKILL.md line 33: { capabilities: { tools: {} } } — that object is shipped in the server's initialize response, and it is the only place a capability declaration appears in this example.
2 — A server ships { capabilities: { tools: {} } } at initialize. The client then sends a resources/list request. What is the correct behaviour?
The server never declared resources. Non-declared methods are out of contract, so the correct answer is to refuse with a JSON-RPC error rather than fabricate an empty success.
3 — Why is capabilities negotiated at initialize rather than per-request?
Per-request renegotiation would force both ends to re-agree on every frame. The handshake collapses that cost to a single round-trip and the rest of the session runs inside that agreement.
