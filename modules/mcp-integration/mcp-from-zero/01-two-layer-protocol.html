The Two-Layer Protocol

// Tier: Adept · Source: meta-prompting-plugin/docs/theory/foundations.md

01Primer

Transport carries bytes. Protocol carries meaning.

MCP is built from two layers stacked on top of each other, and the whole design pays off because that split is clean. The bottom layer is a transport — stdio pipes, a WebSocket, an HTTP stream — whose only job is to move opaque byte frames between a client process and a server process. The top layer is a message protocol — JSON-RPC 2.0 — whose only job is to turn those bytes into typed request / response / notification objects with method names, parameters, and ids.

The meta-prompting-plugin's docs/theory/foundations.md uses a lens that is almost perfect for this: it treats every system as four primary objects — Task, Prompt, Context, and a Quality q ∈ [0,1] scalar — connected by structure-preserving morphisms. Map that onto MCP and the split is obvious. The transport is a functor Bytes → Frames: it preserves order and identity, nothing more. The protocol is a monad Frame →ⁿ Frame: it chains requests, tracks ids, threads conversational state, and gates everything on well-typed payloads.

Why does the split matter? Because you can swap transports without rewriting a single handler. StdioServerTransport and a WebSocket transport both feed the same JSON-RPC layer the same JSON frames. Your listTools handler does not know — and does not need to know — whether its frame arrived over a pipe or a socket. Two layers; one responsibility each; any real MCP server you read will make this separation load-bearing.

Transport is a functor over bytes; JSON-RPC is a monad over frames. Keep them separate and any transport plugs into any server.

## The primary objects

- **Task `T`** — a goal in natural language: "implement X", "summarize Y".
- **Prompt `P`** — the string we actually send to the model.
- **Context `C`** — the surrounding state: history, code, tool outputs.
- **Quality `q ∈ [0,1]`** — a scalar we track alongside every transformation.

## The primary morphisms (commands)

### `/meta` — Functor `F : Task → Prompt`

A functor is a structure-preserving map. `/meta` takes a task and produces a prompt such that:

- **Identity:** `F(id_T) = id_P` — the "do nothing" task maps to the "do nothing" prompt.
- **Composition:** `F(g ∘ f) = F(g) ∘ F(f)` — composing two tasks first and then mapping equals mapping each and then composing.

In practice: complexity classification is a functor. Tier `L3` tasks always produce `L3`-shaped prompts.

### `/rmp` — Monad `M : Prompt →ⁿ Prompt`

A monad is a way to chain computations that carry extra context. `M = (Prompt, return, bind)`:

- `return : Prompt → M(Prompt)` — wrap a prompt in quality-tracking context.
- `bind : M(Prompt) × (Prompt → M(Prompt)) → M(Prompt)` — refine, carrying quality forward.

05Self-Check

1 — MCP splits into two layers. Which pair correctly identifies them?
MCP is a transport layer (moves bytes) plus a JSON-RPC 2.0 message layer (moves typed meaning). The categorical reading in docs/theory/foundations.md calls transport a structure-preserving functor and the protocol a monad that chains typed frames.
2 — In the foundations.md morphism table, what structure is /rmp modelled as?
docs/theory/foundations.md line 28: ### /rmp — Monad M : Prompt →ⁿ Prompt. /rmp iterates refine >=> refine >=> ... with quality carried forward — the same shape MCP's JSON-RPC layer uses when chaining request/response frames.
3 — Why can an MCP server swap StdioServerTransport for a WebSocket transport without changing its tool handlers?
Handler code is layer-above. The transport preserves message shape; the monadic request/response layer above it stays intact. Swapping the carrier does not change the cargo.
