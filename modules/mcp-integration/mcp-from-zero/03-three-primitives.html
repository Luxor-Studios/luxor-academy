01Primer

Three primitives, three categorical shapes

MCP has exactly three primitives that server authors implement, and the mcp-categorical skill encodes each one as a distinct categorical shape. Once you see the shape, the three stop feeling like three arbitrary API families and start feeling like three slots on a single interface.

Tools are morphisms. A tool is a function Parameters → Result. It has a schema on the input side, it returns content on the output side, and it is the only primitive that does something. Every tool handler is a morphism in the category of tools; composing two tools is composing two functions.

Resources are objects. A resource is addressable by URI and has a read() that returns bytes or text. Resources do not act; they exist. They are the objects that tool morphisms consume and produce. A resource with a stable URI is a named point in the category; without the URI it would just be data in motion.

Prompts are exponentials. A prompt template takes a Context and returns a rendered prompt string: Context → String. In category-theory vocabulary that is an exponential object String^Context — a first-class function you can pass around, compose, and parameterise. The composePrompts helper in the skill is exactly the composition of two exponentials.

Hold those three shapes in mind and every MCP design question becomes sharper. Is this a thing I invoke? Tool. Is this a thing I address? Resource. Is this a parameterised render of context? Prompt.

Tools are morphisms, resources are objects, prompts are exponentials. Three primitives; three categorical slots; no overlap.

# MCP Categorical Tool Composition

Categorical patterns for Model Context Protocol server design and tool composition.

## Core Categorical Mapping

MCP structures map to category theory:

- **Tools**: Morphisms `Parameters → Result`
- **Resources**: Objects with URI-based identity
- **Prompts**: Exponential objects `Context → Prompt`
- **Context**: Product type of available information
- **Server**: Category of tools with composition

05Self-Check

1 — Per mcp-categorical/SKILL.md §Core Categorical Mapping, a Tool is formally which categorical construct?
SKILL.md line 14: **Tools**: Morphisms Parameters → Result. A tool is a function from a typed parameter object to a typed result.
2 — The skill maps a Resource to an object with URI-based identity. Why URI rather than numeric id?
The ResourceCategory registry in SKILL.md keys on URI precisely because URIs are stable across servers and sessions — two servers can expose the same logical resource at the same URI without collision.
3 — Prompts are modelled as exponential objects Context → Prompt. What does that let you do that a plain string template could not?
Exponentials are first-class morphisms: you can compose, pass, partially apply, and reason about them. A plain template is inert data; an exponential is a morphism in the category.
