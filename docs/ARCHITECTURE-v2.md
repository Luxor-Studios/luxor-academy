# Architecture v2 — Shadcn/UI Pivot Decision Record

**Date:** 2026-04-19
**Status:** Accepted
**Extends:** `.planning/launch-plan-v1.1.md` (v1.1 remains authoritative for pedagogical + slot contract invariants)
**Supersedes:** nothing

---

## Context

Manu re-scoped the academy from "Phase 1 BARQUE flagship Quest" to a three-tier curriculum (Novice / Experienced / Expert) that teaches LUXOR's full methodology. Two new directives also landed:

1. **Use shadcn/ui** for repeatable, customizable workflows with high-WOW-factor visuals and interactivity.
2. **Use the `codebase-to-course` skill extensively** as the content generation primary tool.

These directives appear to conflict with v1.1 invariants that read:

- "no SSR, no WebGL" (MARS systems validation)
- "one renderer tree; `data-zoom` switches Atlas/Quest/Module" (MERCURIO pedagogical validation)
- "tokens-only theming (no runtime theme JS)" (both)

## Decision

**Adopt a two-layer architecture with a shared design token system.**

### Layer 1 — Shell (`web/` subdirectory)

- **Next.js 15 App Router** with **static export** (`output: 'export'`)
- **shadcn/ui** component library (New York style for ceremonial density)
- **Tailwind v4** with `@theme` directive importing `atlas/tokens/tokens.css` values
- **Scope:** landing page, tier pages (`/novice`, `/experienced`, `/expert`), progress dashboard, Atlas hub, XP sovereignty UI, global navigation

**Why this honors the "no SSR" invariant:** static export produces plain HTML + JS assets at build time. No runtime Node server. Deploys as static files to any CDN. The invariant was about runtime rendering complexity, not about the build tool.

**Why this enables shadcn/ui:** shadcn/ui is distributed as copy-paste React components. Static export preserves the component composition ergonomics at development time while shipping static HTML to users.

### Layer 2 — Modules (`modules/<track>/<quest>/*.html`)

- Self-contained HTML files generated by the `codebase-to-course` skill
- One file per module, offline-capable, no framework runtime
- Validated against `validators/slots.schema.json` (the 6-slot contract)
- Linked from the shell; can be opened standalone
- Rendered via the existing atlas web-component tree OR the skill's own renderer

**Why this honors "SVG/CSS only / no WebGL":** the skill's output is exactly this. Animations are CSS. Visualizations are SVG. Interactivity is vanilla JS sandboxed per the slot contract.

### Shared design tokens

`atlas/tokens/tokens.css` is the single source. Tailwind v4 imports it in the shell's `globals.css`:

```css
@import "tailwindcss";
@import "../../atlas/tokens/tokens.css";

@theme {
  --color-gold: var(--luxor-gold);
  --color-navy: var(--luxor-navy);
  --font-display: var(--font-playfair);
  --font-body: var(--font-inter);
  --font-mono: var(--font-jetbrains);
}
```

Modules load `atlas/tokens/tokens.css` directly. Same values, two consumers. Change the token, both layers update.

## Consequences

### What stays the same (v1.1 invariants honored)

1. ✅ Tokens-only theming — Tailwind reads tokens, does not define new ones
2. ✅ Three fonts max — Playfair / Inter / JetBrains
3. ✅ 6-slot module contract (`validators/slots.schema.json`)
4. ✅ SVG/CSS only in modules (codebase-to-course output shape)
5. ✅ Sandbox policy per widget
6. ✅ XP export mandatory (now a shadcn/ui `<Dialog>` in the shell)
7. ✅ `ship_your_own` canonical last module
8. ✅ Every technical claim cited to a file or official doc

### What changes from v1.1

1. Atlas is no longer the ONLY shell surface. It becomes ONE view inside the larger shell (route: `/atlas`).
2. `data-zoom` web-component still runs on the Atlas route. Tier pages and the dashboard are React/shadcn components instead.
3. Build toolchain now includes Next.js (previously: vanilla HTML + node test scripts only). Tradeoff: +50MB node_modules in `web/` for the shell, vs. qualitative improvement in UX velocity and component reuse.
4. Two Lighthouse budget scopes: module pages still target interactive ≤ 1000ms; shell pages target interactive ≤ 2000ms (a richer shell is allowed more bytes).

### What the shell explicitly does NOT do

- No server-rendered pages at runtime (static export only)
- No auth, no user accounts (deferred to Phase 5)
- No XP persistence to a server (localStorage + export button)
- No CMS — tier/track/quest data lives in TypeScript files committed to the repo
- No analytics SDK (privacy: defer to Phase 2's anonymous-XP work)

## Team composition

This architecture is built by an agentic team launched on demand via the `Agent` tool:

| Role | Tool |
|------|------|
| Main orchestrator | this session |
| Content generator | `codebase-to-course` skill invoked by subagents |
| Three-plane verifier | `mercurio-orchestrator` agent |
| Systems verifier | `MARS` agent |
| Deep project research | `deep-researcher` / `code-explorer` agent |
| Frontend component work | `frontend-architect` agent |

No elaborate orchestration machinery. Each subagent is launched for a specific task, returns a result, exits. The main session stays lean.

## Verification plan

Every tier page and every generated module passes through:

1. `npm run validate:fixtures` — slot-contract regression check stays green
2. `npm run test:e2e` — Playwright smoke across the full shell + module flow
3. `npm run test:a11y` — axe-core zero violations on rendered HTML
4. `npm run test:perf` — Lighthouse budgets per scope
5. **MERCURIO** three-plane coherence review (Mental / Physical / Spiritual)
6. **MARS** systems-level architectural validation

Gates: MERCURIO ≥ 8/10 per plane, MARS ≥ 85% confidence. Otherwise halt for human review.
